# docs 目录文档说明及开发提示词

本文档用于说明 `docs` 目录下各个 Markdown 文件的作用，方便快速了解项目文档分工。

## 建议阅读顺序

1. `docs/SPEC.md`
2. `docs/PLAN.md`
3. `docs/STATUS.md`
4. `docs/HANDOFF.md`
5. `docs/DECISIONS.md`
6. `docs/IMPLEMENTATION_PLAN.md`
7. `docs/API_CONTRACT.md`
8. `docs/DB_MODEL_DRAFT.md`
9. `docs/AGENTS.md`

## 各文档作用

### `docs/SPEC.md`

项目规格说明。

作用：
- 定义一期项目目标、非目标和边界
- 明确左右双屏、后台管理、数据接入等核心范围
- 统一业务规则、关键页面、数据对象和技术约束
- 作为研发实现时的主要规格依据

适合什么时候看：
- 开始开发新功能前
- 判断某项需求是否属于一期范围时
- 确认业务规则和验收边界时

### `docs/PLAN.md`

项目里程碑计划。

作用：
- 将项目拆分为 M0 到 M9 的阶段
- 明确每个阶段的目标、任务、验收标准和依赖关系
- 帮助判断当前工作应该落在哪个里程碑内

适合什么时候看：
- 安排下一轮开发任务前
- 判断当前是不是跨阶段开发时
- 做阶段性排期和交付规划时

### `docs/STATUS.md`

当前项目状态记录。

作用：
- 记录当前处于哪个阶段
- 汇总当前已完成内容、已知约束、未确认事项和下一步行动
- 作为“项目现在进行到哪里”的快速入口

适合什么时候看：
- 接手项目时
- 每轮开发开始前
- 想快速了解当前最新进展时

### `docs/HANDOFF.md`

交接文档。

作用：
- 给下一位继续开发的人提供上下文
- 汇总已确认决策、未完成事项、注意点和建议下一步
- 记录每一轮开发结束后的交接说明

适合什么时候看：
- 新代理或新成员接手时
- 每轮开发结束后回顾当前交付时
- 想了解为什么之前这样实现时

### `docs/DECISIONS.md`

技术决策记录。

作用：
- 记录开发过程中“为什么这么做”
- 补充规格文档和交接文档里不适合展开的实现背景
- 沉淀认证、权限、接口风格、缓存、安全、阶段边界等关键决策

适合什么时候看：
- 想知道当前实现背后的原因时
- 准备调整已有方案前
- 评估某项改动是否会推翻既有决策时

### `docs/IMPLEMENTATION_PLAN.md`

实施级开发计划。

作用：
- 比 `PLAN.md` 更偏执行层
- 用于拆解具体开发任务、依赖、风险和推荐推进顺序
- 帮助把里程碑落成实际工作项

适合什么时候看：
- 进入具体编码实施前
- 拆开发任务时
- 评估某块工作是否可以并行推进时

### `docs/API_CONTRACT.md`

接口契约草案。

作用：
- 定义后端 API 的结构、字段和返回风格
- 约束前后端之间的数据交互格式
- 避免前端直接依赖外部系统原始结构

适合什么时候看：
- 新增或调整 API 前
- 前端接接口前
- 检查字段命名和响应格式是否一致时

### `docs/DB_MODEL_DRAFT.md`

数据库模型草案。

作用：
- 描述一期涉及的数据对象和模型设计方向
- 帮助统一主数据、映射、缓存和配置类数据的结构认识
- 作为落地 Django 模型或数据库设计时的参考

适合什么时候看：
- 设计或修改数据模型前
- 落库前评审数据结构时
- 判断哪些字段可以先 mock、哪些要等外部系统资料时

### `docs/AGENTS.md`

代理工作规则。

作用：
- 规定开发代理在本项目中的工作纪律
- 包括需求澄清、文档维护、分阶段实施和变更记录要求
- 约束代理不要跳阶段、不要脱离文档直接扩范围

适合什么时候看：
- 代理开始工作前
- 检查当前工作方式是否符合项目规则时

## 简单分工总结

- `SPEC.md`：定义做什么
- `PLAN.md`：定义按什么阶段做
- `STATUS.md`：记录现在做到哪里
- `HANDOFF.md`：告诉下一位怎么接着做
- `DECISIONS.md`：记录为什么这样做
- `IMPLEMENTATION_PLAN.md`：把阶段计划拆成执行任务
- `API_CONTRACT.md`：约束接口怎么设计
- `DB_MODEL_DRAFT.md`：约束数据模型怎么设计
- `AGENTS.md`：约束代理怎么工作

## AI开发技巧

#### 每个新会话先发提示词

你现在是这个项目的持续开发代理。

开始前必须先阅读以下文件，并把它们视为当前项目的唯一事实来源：
1. PRD_和泰智屏系统.md
2. SPEC.md
3. PLAN.md
4. STATUS.md
5. HANDOFF.md
6. AGENTS.md
7. DECISIONS.md

工作规则：
- 严格按文档执行，不要擅自扩大范围。
- 一期前段只做外部参观双屏大屏，不包含内部 Web 报表。
- 报修与 3D 仿真不是一期前段阻塞项。
- 前端不得直连外部系统。
- 后端负责定时拉取、标准化、缓存和对前端提供 API。
- 数据源异常时，大屏必须显示最近一次成功数据，不允许白屏。
- 每一轮只完成一个明确里程碑或一个最小可交付目标。
- 每完成一轮后，必须更新 STATUS.md 和 HANDOFF.md。
- 不要无必要重构。
- 所有改动后必须给出验证步骤和结果。

先输出：
1. 你理解的当前项目状态
2. 当前最合理的下一步
3. 本轮不做什么
4. 本轮完成标准

得到这些结论后，再开始执行。

#### 每轮开发完毕后提示词

本轮开发完成后，不要直接结束。

请必须完成以下收尾动作：
1. 更新 STATUS.md
   - 当前阶段
   - 本轮已完成
   - 未完成
   - 当前已知问题
   - 下一个最优先任务

2. 更新 HANDOFF.md
   - 本轮目标
   - 本轮实际完成
   - 本轮未完成
   - 修改文件清单
   - 遇到的问题
   - 建议下一轮优先任务

3. 更新DECISIONS.md

   更新开发中所有的决策、理由、影响。

5. 输出最终总结：

   - 本轮完成内容

   - 修改了哪些文件

   - 如何验证

   - 是否存在阻塞

   - 下一轮建议
