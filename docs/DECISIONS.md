# 技术决策 / DECISIONS

本文档记录开发过程中“为什么这么做”，用于补充 `SPEC.md`、`PLAN.md`、`STATUS.md`、`HANDOFF.md` 之外的实现决策背景。

---

## 2026-04-13

### 决策：一期范围先只做外部参观双屏大屏，不把内部 Web 报表纳入一期
原因：
- PRD 和 SPEC 都明确一期聚焦外部参观场景
- 内部报表需求、权限和统计口径尚未稳定
- 先收敛范围，避免一期目标被持续扩张

影响：
- 一期开发不进入内部报表页面和报表 API
- 二期需要单独再做 PRD、模型和权限规划

### 决策：报修与 3D 仿真不作为一期前段阻塞项
原因：
- 报修系统取数条件不稳定
- 3D 仿真优先级低于一期核心展示链路
- 一期验收关键仍是设备、产量、排产、能耗稳定展示

影响：
- 一期前段只保留报修占位和 3D 预留区
- 报修和 3D 的真实接入放到后续阶段

### 决策：前端不直连外部系统，统一由后端拉取、标准化、缓存后提供 API
原因：
- 可屏蔽 SAP、排产库、能耗库、OPCUA 等外部系统差异
- 有利于统一鉴权、错误处理和缓存策略
- 满足“数据源异常时大屏显示最近一次成功数据”的要求

影响：
- 前端只依赖本系统 API，不依赖外部系统结构
- 后端需要承担定时拉取、标准化、缓存和健康状态管理

## 2026-04-14

### 决策：后台权限一期先只做管理员权限，不做复杂 RBAC
原因：
- 当前阶段目标是尽快具备最小后台维护能力
- 一期只有后台管理场景，复杂角色拆分收益有限
- 可以降低模型、接口和前端管理复杂度

影响：
- 当前后台接口统一要求管理员身份
- 后续如果进入多角色体系，需要在现有接口上补权限层

### 决策：管理员认证当前采用 Django 签名 Bearer Token，而不是直接引入 JWT
原因：
- 当前是 M2 最小后台能力，优先保证实现简单、可用、依赖少
- 可直接复用 Django 用户体系和签名能力
- 当前还没有网关、刷新令牌、多端会话等复杂诉求

影响：
- 前端仍要处理登录态过期和重新登录
- 如果后续接入统一网关或更复杂认证体系，可能迁移到 JWT 或集中认证方案

### 决策：后台接口统一使用 `success/code/message/data` 响应包裹格式
原因：
- 便于前后端统一处理成功和失败场景
- 能减少不同模块返回风格不一致的问题
- 有利于后续继续扩展管理接口和展示接口

影响：
- 新增 API 时应保持同样结构
- 需要统一异常处理，避免直接漏出框架默认错误格式

### 决策：数据源敏感配置采用“结构预留”方式，而不是当前阶段直接落地真实加密方案
原因：
- 一期当前还未接真实外部系统，真实密钥流转方案尚未最终确认
- 但数据模型必须提前体现环境变量引用和密文存储两类能力
- 可以先保证接口与模型不会把敏感信息写死在代码里

影响：
- 当前 `DataSourceConfig` 支持 `env_ref` 和 `encrypted` 两种结构
- 真正的密文加解密机制、密钥轮换和运维流程后续仍需补齐

### 决策：M2 先提供最基础的后台前端界面，而不是继续只堆 API
原因：
- 只有 API 而没有最小界面，后台能力无法被直接使用和验证
- 最小界面可以更早暴露字段设计和接口设计问题
- 有助于后续 M3/M4 开发时直接维护配置和 mock 数据

影响：
- 当前已提供 `/admin/login` 和 `/admin/console`
- 该界面以可用为主，不做复杂交互和视觉包装

### 决策：根据数据库结构审阅结果补充员工表，并将员工角色先收敛为固定枚举
原因：
- 当前数据库结构缺少员工台账，不利于后续后台维护和业务扩展
- 本轮需求已经明确员工号格式和角色范围，无需等待更多外部系统资料
- 一期仍应保持简单，不引入复杂组织架构、岗位体系或权限继承

影响：
- 后台新增员工台账模型、API 和前端管理入口
- 员工号当前限制为英文和数字，禁止特殊字符
- 角色当前固定为 `employee`、`team_leader`、`supervisor`，后续如扩展需评估兼容性
## 2026-04-14（员工台账补充收尾）

### 决策：员工号当前仅允许英文和数字
原因：
- 这是本轮用户明确提出的输入约束。
- 先收紧格式，可以减少后续与外部编码、条码或工号体系对齐时的歧义。
- 当前阶段只做最小后台能力，不引入更复杂的编号兼容逻辑。

影响：
- 后端对 `employee_no` 增加正则校验，禁止空格、连字符和其他特殊字符。
- 前端员工台账录入需要遵循同样约束。
- 如果后续现场存在带分隔符的工号，需要评估兼容策略并安排数据迁移。

### 决策：员工角色先收敛为固定枚举，不做可配置角色体系
原因：
- 当前一期后台只做管理员权限，不做复杂 RBAC。
- 本轮只需满足员工、班组长、主管三类角色展示和维护。
- 先固定枚举可以保持 API、前端表单和数据库结构简单一致。

影响：
- 当前角色值固定为 `employee`、`team_leader`、`supervisor`。
- 前后端均按固定选项实现，不开放后台自定义角色。
- 若后续出现更多岗位类型，需要评估是否升级为独立角色主数据。

### 决策：员工台账先作为独立主数据维护，不与后台登录账号绑定
原因：
- 本轮目标是补足数据库结构中的缺口，而不是扩展认证或人员体系。
- 员工主数据与管理员登录账号并不是同一问题域，强行绑定会扩大范围。
- 这样可以保持本轮只完成一个明确的最小可交付目标。

影响：
- `Employee` 只承载台账信息，不参与当前管理员登录认证。
- 后续如需做员工自助登录、审批链或组织关系，再单独设计关联模型。

## 2026-04-17（M2 最终收尾）

### 决策：新增 Material 和 Order 独立主数据模型，而不是在 Device 或其他现有表上扩展
原因：
- PRD 和 DOCS_OVERVIEW 在 M2 规划中明确需要物料和订单台账能力。
- 物料和订单有独立的业务实体和字段集，不应混入设备或产线表。
- 独立模型有利于后续 M3/M4 将订单和物料与排产数据、展示数据关联。

影响：
- 新增 `Material` 和 `Order` 两张独立表，均继承 `ReservedFieldsMixin` 和 `TimestampedModel`。
- Order 通过 `ForeignKey(PROTECT)` 关联 Material 和 ProductionLine，删除物料或产线前必须先处理关联订单。
- 后续如需支持多产线订单（一个订单分配到多条产线），需将单产线 FK 改为多对多关系。

### 决策：Order 与 ProductionLine 的关系当前采用单产线 FK，不做多对多
原因：
- 当前尚未确认一个订单是否会同时分配到多条产线。
- 先采用最简单的单产线 FK，降低模型和前端表单复杂度。
- 如果后续确认需要多产线，可以通过新增中间表升级。

影响：
- 当前 `Order.production_line` 为可空 FK，允许暂不指定产线。
- 前端表单中产线选择为单选下拉框。
- 如后续需要多对多，需数据迁移和前端表单改造。

### 决策：Order 的状态字段采用固定四值枚举，不做可配置状态流
原因：
- 当前阶段订单台账仅作为主数据维护，不涉及工作流或审批。
- 四种状态（计划、生产中、已完成、已取消）已覆盖订单全生命周期基本节点。
- 先固定枚举可以保持 API、前端表单和数据库结构简单一致。

影响：
- `Order.status` 固定为 `planned`、`in_progress`、`completed`、`cancelled`。
- 前后端均按固定选项实现，不开放后台自定义状态。
- 如后续出现更复杂工单流转，需评估是否引入状态机或独立工序模型。

### 决策：将 PageModuleSwitch 拆分为独立模型，不再依赖 ScreenConfig.module_settings JSON 字段
原因：
- ScreenConfig.module_settings 是一个 JSON 字段，缺乏结构化校验和独立的 CRUD 能力。
- 页面模块开关是一个高频管理操作（启用/禁用某个展示模块），需要明确的接口和权限控制。
- 独立模型支持联合唯一约束（screen_key + module_key），可以从数据库层面防止配置冲突。

影响：
- 新增 `PageModuleSwitch` 表，screen_key + module_key 联合唯一。
- ScreenConfig 中的 `module_settings` JSON 字段仍保留但不再作为模块开关的正式来源。
- 后续 M3/M4 中前端展示接口应优先读取 PageModuleSwitch 表。

### 决策：所有新增业务模型继续继承 ReservedFieldsMixin，保持预留字段一致
原因：
- 上一轮已确定 9 个既有业务模型均具备 reserved_1 到 reserved_5 预留字段。
- 新增的 Material、Order、PageModuleSwitch 应保持一致的扩展性。
- 统一继承可以减少后续补字段的迁移成本。

影响：
- Material、Order、PageModuleSwitch 均包含 reserved_1 到 reserved_5 字段。
- 前端编辑表单中均可维护预留字段。

### 决策：M2 阶段以正式 MySQL 执行全量冒烟回归，不仅依赖 SQLite 测试
原因：
- 此前所有测试均基于 `hota_mds.test_settings`（SQLite），无法验证 MySQL 特有的行为（如字符集、排序规则、外键约束执行差异）。
- M2 作为后台能力基线，需要在正式数据库配置上验证一次完整链路后才能确认可交付。
- 冒烟回归覆盖登录、所有台账 CRUD、所有配置 CRUD、操作日志和健康检查，确保不遗漏。

影响：
- 已验证 20/20 冒烟检查通过，包括 Area、ProductionLine、Device、Employee、Material、Order、CodeMapping、ScreenConfig、PageModuleSwitch、DisplayContentConfig、RuntimeParameterConfig、DataSourceConfig、OperationLog、HealthCheck。
- 后续 M3 开发可以直接在 MySQL 上运行，不再需要重复验证 M2 基础。
