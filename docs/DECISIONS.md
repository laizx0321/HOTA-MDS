# 技术决策 / DECISIONS

本文档记录开发过程中“为什么这么做”，用于补充 `SPEC.md`、`PLAN.md`、`STATUS.md`、`HANDOFF.md` 之外的实现决策背景。

---

## 2026-04-13

### 决策：一期范围先只做外部参观双屏大屏，不把内部 Web 报表纳入一期
原因：
- PRD 和 SPEC 都明确一期聚焦外部参观场景
- 内部报表需求、权限和统计口径尚未稳定
- 先收敛范围，避免一期目标被持续扩张

影响：
- 一期开发不进入内部报表页面和报表 API
- 二期需要单独再做 PRD、模型和权限规划

### 决策：报修与 3D 仿真不作为一期前段阻塞项
原因：
- 报修系统取数条件不稳定
- 3D 仿真优先级低于一期核心展示链路
- 一期验收关键仍是设备、产量、排产、能耗稳定展示

影响：
- 一期前段只保留报修占位和 3D 预留区
- 报修和 3D 的真实接入放到后续阶段

### 决策：前端不直连外部系统，统一由后端拉取、标准化、缓存后提供 API
原因：
- 可屏蔽 SAP、排产库、能耗库、OPCUA 等外部系统差异
- 有利于统一鉴权、错误处理和缓存策略
- 满足“数据源异常时大屏显示最近一次成功数据”的要求

影响：
- 前端只依赖本系统 API，不依赖外部系统结构
- 后端需要承担定时拉取、标准化、缓存和健康状态管理

## 2026-04-14

### 决策：后台权限一期先只做管理员权限，不做复杂 RBAC
原因：
- 当前阶段目标是尽快具备最小后台维护能力
- 一期只有后台管理场景，复杂角色拆分收益有限
- 可以降低模型、接口和前端管理复杂度

影响：
- 当前后台接口统一要求管理员身份
- 后续如果进入多角色体系，需要在现有接口上补权限层

### 决策：管理员认证当前采用 Django 签名 Bearer Token，而不是直接引入 JWT
原因：
- 当前是 M2 最小后台能力，优先保证实现简单、可用、依赖少
- 可直接复用 Django 用户体系和签名能力
- 当前还没有网关、刷新令牌、多端会话等复杂诉求

影响：
- 前端仍要处理登录态过期和重新登录
- 如果后续接入统一网关或更复杂认证体系，可能迁移到 JWT 或集中认证方案

### 决策：后台接口统一使用 `success/code/message/data` 响应包裹格式
原因：
- 便于前后端统一处理成功和失败场景
- 能减少不同模块返回风格不一致的问题
- 有利于后续继续扩展管理接口和展示接口

影响：
- 新增 API 时应保持同样结构
- 需要统一异常处理，避免直接漏出框架默认错误格式

### 决策：数据源敏感配置采用“结构预留”方式，而不是当前阶段直接落地真实加密方案
原因：
- 一期当前还未接真实外部系统，真实密钥流转方案尚未最终确认
- 但数据模型必须提前体现环境变量引用和密文存储两类能力
- 可以先保证接口与模型不会把敏感信息写死在代码里

影响：
- 当前 `DataSourceConfig` 支持 `env_ref` 和 `encrypted` 两种结构
- 真正的密文加解密机制、密钥轮换和运维流程后续仍需补齐

### 决策：M2 先提供最基础的后台前端界面，而不是继续只堆 API
原因：
- 只有 API 而没有最小界面，后台能力无法被直接使用和验证
- 最小界面可以更早暴露字段设计和接口设计问题
- 有助于后续 M3/M4 开发时直接维护配置和 mock 数据

影响：
- 当前已提供 `/admin/login` 和 `/admin/console`
- 该界面以可用为主，不做复杂交互和视觉包装

### 决策：根据数据库结构审阅结果补充员工表，并将员工角色先收敛为固定枚举
原因：
- 当前数据库结构缺少员工台账，不利于后续后台维护和业务扩展
- 本轮需求已经明确员工号格式和角色范围，无需等待更多外部系统资料
- 一期仍应保持简单，不引入复杂组织架构、岗位体系或权限继承

影响：
- 后台新增员工台账模型、API 和前端管理入口
- 员工号当前限制为英文和数字，禁止特殊字符
- 角色当前固定为 `employee`、`team_leader`、`supervisor`，后续如扩展需评估兼容性
## 2026-04-14（员工台账补充收尾）

### 决策：员工号当前仅允许英文和数字
原因：
- 这是本轮用户明确提出的输入约束。
- 先收紧格式，可以减少后续与外部编码、条码或工号体系对齐时的歧义。
- 当前阶段只做最小后台能力，不引入更复杂的编号兼容逻辑。

影响：
- 后端对 `employee_no` 增加正则校验，禁止空格、连字符和其他特殊字符。
- 前端员工台账录入需要遵循同样约束。
- 如果后续现场存在带分隔符的工号，需要评估兼容策略并安排数据迁移。

### 决策：员工角色先收敛为固定枚举，不做可配置角色体系
原因：
- 当前一期后台只做管理员权限，不做复杂 RBAC。
- 本轮只需满足员工、班组长、主管三类角色展示和维护。
- 先固定枚举可以保持 API、前端表单和数据库结构简单一致。

影响：
- 当前角色值固定为 `employee`、`team_leader`、`supervisor`。
- 前后端均按固定选项实现，不开放后台自定义角色。
- 若后续出现更多岗位类型，需要评估是否升级为独立角色主数据。

### 决策：员工台账先作为独立主数据维护，不与后台登录账号绑定
原因：
- 本轮目标是补足数据库结构中的缺口，而不是扩展认证或人员体系。
- 员工主数据与管理员登录账号并不是同一问题域，强行绑定会扩大范围。
- 这样可以保持本轮只完成一个明确的最小可交付目标。

影响：
- `Employee` 只承载台账信息，不参与当前管理员登录认证。
- 后续如需做员工自助登录、审批链或组织关系，再单独设计关联模型。

## 2026-04-20

### 决策：M3 先用标准化缓存快照 + mock 装载命令跑通展示接口，不提前接真实数据源
原因：
- 当前外部系统点表、字段和连接方式仍未到位，但前后端都需要稳定的标准展示接口。
- PRD、SPEC、PLAN、STATUS、HANDOFF 都要求当前阶段不要让前端直连外部系统，并优先保证“最近一次成功数据兜底”。
- 先把缓存快照、mock 装载和展示 API 跑通，可以在不扩大范围的前提下验证标准模型与异常策略。

影响：
- 当前左右屏接口 `/api/screens/left`、`/api/screens/right` 先读取标准化快照，而不是读取任何外部系统原始结构。
- 当前提供 `load_mock_screen_data` 管理命令，用于装载 mock 快照和模拟失败场景。
- 后续进入 M5 时，真实采集任务应复用这批标准快照模型写入数据，而不是绕过缓存层直接改大屏接口结构。

### 决策：展示配置中的历史脏文案优先在后端读取层做兜底，而不是让前端分散处理
原因：
- 当前已有历史坏值进入 `DisplayContentConfig`，典型表现为公司名或欢迎语只剩 `?`。
- 这类问题属于展示数据质量问题，应该由后端在标准化输出阶段统一兜底，而不是让多个前端页面分别判断。
- 后端统一兜底更符合“后端负责标准化、前端负责展示”的项目规则。

影响：
- 当前左/右屏接口在读取展示内容配置时，会自动把仅由问号组成的坏值回退为默认中文文案。
- 前端页面不需要再为这类历史脏值增加重复判断逻辑。
- 后续若出现更多展示类脏值，应优先继续补在后端标准化层，而不是在具体页面修补。

### 决策：双屏中的状态枚举和占位说明继续下沉到后端标准化输出，前端只消费展示字段
原因：
- 左屏 `running/stopped/alarm/offline`、右屏 `normal/warning/delayed/paused` 这类值本质上属于内部状态键，不适合直接暴露给外部参观者。
- 如果由前端自行维护这些映射，后续多个页面或多个展示端容易出现文案不一致。
- 当前项目规则明确要求后端负责标准化、缓存并向前端提供 API。

影响：
- 左屏接口新增 `deviceOverview.statusItems`，右屏接口新增 `schedule.riskSummary.items`，统一提供可直接展示的 `label`、`count`、`accent` 等字段。
- 前端 `ScreenDisplay` 改为直接渲染这些标准化字段，不再自己解释业务枚举。
- 后续继续扩展双屏字段时，应优先沿用这种“后端给展示语义，前端只负责布局渲染”的模式。

### 决策：报修与 3D 在一期仍保持非阻塞占位，但占位说明也由后端统一提供
原因：
- 报修与 3D 仿真当前仍不是一期前段阻塞项，但展示页需要明确说明其阶段定位，避免参观时产生误解。
- 如果占位说明写死在前端，后续阶段描述调整时需要多处同步修改。
- 将占位说明也纳入接口返回，可以保持左右屏展示语义统一可控。

影响：
- 当前接口新增 `repairPlaceholder.description` 与 `simulationPlaceholder.description`。
- 前端不再硬编码这两块说明文案。
- 后续如果项目阶段定位变化，只需调整后端输出或配置来源，不需要分别修改每个展示页。

## 2026-04-21

### 决策：一期双屏所有对外可见的动态展示语义继续统一由后端输出 `display` 字段，前端只负责渲染
原因：
- 当前 M3 收尾的剩余问题，主要就是少量可见字段仍由前端自行格式化，如最近成功时间、状态计数、趋势标签、窗口文案和风险计数。
- 如果这类字段继续散落在前端拼接，会削弱“后端负责标准化、前端负责展示”的项目边界。
- 对外参观场景更强调展示一致性和稳定性，统一从后端输出更容易管控口径。

影响：
- 当前新增或补齐了 `meta.display.lastSuccessfulAtLabel`、`deviceOverview.statusItems[].countLabel`、`productionTrend[].display`、`schedule.display.windowDaysLabel`、`schedule.riskSummary.items[].countLabel`。
- 前端 `ScreenDisplay` 进一步收敛为布局和渲染层，不再承担这些文案与格式拼装职责。
- 后续双屏新增展示字段时，应优先在后端标准化层增加 `display` 语义，而不是先在前端临时拼接。

### 决策：在标准化输出扩展时保留原始业务字段，避免为 M3 收尾做无必要重构
原因：
- 本轮目标是完成 M3 收尾和口径统一，而不是重写快照结构。
- 现有测试、mock 数据和前端回退逻辑都仍依赖部分原始字段，直接移除会扩大变更面。
- 用户明确要求不要无必要重构。

影响：
- 当前仍保留 `lastSuccessfulAt`、`count`、`hourLabel`、`producedQuantity`、`windowDays` 等原始字段。
- 新增的 `display` 字段作为展示语义层叠加在现有结构上，兼容已有调用方。
- 后续若要精简原始字段，应放到单独轮次评估，而不是夹带在当前阶段收尾中完成。

### 决策：M3 完成判定以 PLAN 中的正式验收口径为准，而不是继续无限细化局部展示字段
原因：
- `docs/PLAN.md` 对 M3 的验收标准已经明确：mock API、前端可基于 API 开发、缓存读取接口、后台可看 mock 数据源健康状态。
- 当前这些正式条件已经满足，本轮剩余工作只是把展示语义进一步收口，以消除 `STATUS/HANDOFF` 中记录的尾项。
- 如果继续把 M3 无限延长，会导致里程碑边界失真，也会阻塞 M4 的正常推进。

影响：
- 当前项目状态已从 “M3 进行中” 更新为 “M3 已完成，可进入 M4”。
- 下一轮应正式进入 M4，优先处理左右屏轮播、全屏适配和独立配置读取。
- 真实外部系统接入仍保持在 M5，不因 M3 完成而前移。

## 2026-04-22

### 决策：M4 自动轮播先定义为“按 `screen.pageKeys` 在前端预置页面视图间轮播”，而不是新增一套独立轮播接口协议
原因：
- 当前后台配置与屏幕 API 已经具备 `pageKeys` 和 `rotationIntervalSeconds`，但并没有新的页面编排协议。
- 本轮目标是完成一期前段双屏页面，不应为了轮播单独扩展后端接口或引入无必要重构。
- 采用前端预置视图页签的方式，可以在不破坏现有 mock/cache API 的前提下落地自动轮播与左右屏独立配置读取。

影响：
- 左屏当前支持基于配置在 `overview`、`operations`、`energy` 等预置视图间轮播。
- 右屏当前支持基于配置在 `schedule`、`risk`、`simulation` 等预置视图间轮播。
- 若配置仅包含单页键，则页面保持单页展示，不强制轮播。

### 决策：全屏能力采用“用户触发进入全屏”方式，而不是页面加载即自动全屏
原因：
- 浏览器安全策略通常不允许页面在加载完成后无交互直接进入全屏。
- 项目要求是“支持全屏展示”，并未要求绕过浏览器限制做强制自动全屏。
- 提供显式全屏按钮和双击画面切换，可以满足拼接屏展示场景，同时避免兼容性问题。

影响：
- 当前左右屏页面均提供“进入全屏/退出全屏”按钮。
- 当前左右屏页面均支持双击画面切换全屏状态。
- 后续如现场浏览器允许额外自动化能力，应在单独轮次评估，不在本轮扩大实现范围。

### 决策：右屏甘特图任务状态只通过颜色区分，不在任务条上重复显示“风险 / 延期”等文字
原因：
- 当前右屏甘特图的主要阅读目标是快速扫视订单排布与节奏，而不是在每个任务条上阅读完整风险说明。
- 风险文字会挤占短周期任务的有限宽度，直接影响订单号和核心信息的可读性。
- 用户已明确要求“只根据颜色作为区分”。

影响：
- 当前任务条上不再显示“风险 / 延期”等状态文字。
- 风险状态继续通过颜色区分，并由右侧风险说明区域承担解释职责。
- 后续若需要增强状态说明，应优先在图例或侧边摘要中补充，而不是重新塞回任务条内部。

### 决策：右屏甘特图纵向布局优先采用保守安全区，避免任务条在展示环境中被裁切
原因：
- 当前右屏甘特图任务条使用绝对定位叠加，任务内容、边框和内边距会带来实际高度偏差。
- 对外参观大屏最忌出现条块被切掉的视觉缺陷，宁可牺牲少量纵向密度，也要先保证完整显示。
- 用户已多次反馈存在任务块底部被截断问题。

影响：
- 当前任务条纵向槽位、底部安全区和轨道容器留白都采用更保守的计算。
- 当前优先确保任务条完整显示，再在后续现场验收中微调密度。
- 如果后续需要进一步提升密度，应在不破坏完整显示的前提下谨慎调整，必要时再评估改为显式栅格行布局。

### 决策：短周期任务条优先完整显示订单号，允许对窄条使用更小字号和更紧凑内边距
原因：
- 短周期任务在甘特图中的识别优先级最高的是订单号，而不是丰富的二级说明文字。
- 用户明确要求对短周期任务做优化，缩小字体以显示完整订单号。
- 对于参观展示场景，完整订单号比条内次要文案更重要。

影响：
- 当前 `compact` 与 `tiny` 两类短条使用更小字号、更紧字距与更小内边距。
- 当前窄条订单号展示优先级高于省略号观感。
- 后续若现场仍认为极短条不够清晰，可继续只针对极窄条追加更激进的字体策略，而无需改动长条布局。

### 决策：右屏甘特图区滚动条采用“隐藏 UI、保留滚动能力”的处理方式
原因：
- 显式浏览器滚动条会破坏大屏驾驶舱的整体观感，用户已明确表示当前拖动条“太丑，需要隐藏”。
- 自动滚动能力仍是当前右屏验证的重要组成部分，不能因为隐藏滚动条而丢失滚动逻辑。
- 这是纯展示层问题，不需要修改后端接口或前端滚动逻辑。

影响：
- 当前 `.gantt-rows` 的显式滚动条在主流浏览器中会被隐藏。
- 当前自动滚动和手动滚动能力仍然保留。
- 后续若现场设备浏览器对滚动条隐藏兼容性存在差异，应优先在样式层补兼容，而不是撤销该处理。
